# gpt-4o IPS 추출 회귀 평가

평가일: 2026-07-13

모델 응답 버전: `gpt-4o-2024-11-20`

프롬프트 버전: `ips-extract-v2`
설정: `temperature=0`, `seed=42`

## 테스트셋

`tests/fixtures/ips_extraction_cases.json`에 20개 한국어 상담 사례를 고정했다. 이름,
목표 수익 금액, 투자기간, 유동성 등급, 명시적 유동성 필요금액처럼 고객 적합성 판단에
직접 쓰이는 필드만 정답으로 평가한다. Tax·Legal·Unique는 상담 의미를 보존하는 여러 표현이
가능하므로 정확도 정답에는 포함하지 않되, 전체 출력 해시로 변동을 별도 감시한다.

포함한 경계·오류 유발 사례:

- 개월→년, 원→억 원 단위 변환
- 기대수익률(%)과 목표 수익 금액 구분
- 유동성 10%·30% 경계
- 누락된 이름·기간·유동성 금액
- 상담의 나이·직업·위험성향이 고정 시나리오와 충돌하는 입력
- 공백·시장 우려 등 주변 문장이 섞인 입력

## 3회 반복 실측 결과

실행 명령:

```bash
python scripts/evaluate_ips_extraction.py --repeats 3
```

| 지표 | 결과 | 합격 기준 |
| --- | ---: | ---: |
| 사례×반복 정확도 | 100% (60/60) | 95% 이상 |
| 핵심 평가 필드 반복 일치율 | 100% (20/20) | 95% 이상 |
| 자유서술 포함 전체 출력 일치율 | 90% (18/20) | 감사용, 합격 기준 아님 |

동일 최종 평가 안의 `system_fingerprint`는 `fp_9baf9886c5`로 같았다. 다만 그 전에 실행한
그래프에서는 `fp_40854b0bd2`가 관찰되어 Azure 백엔드 구성이 실행 시점 사이에 달라질 수
있음을 확인했다. 따라서 seed는 best-effort 안정화 수단이며 완전 결정론의 증거로 사용하지
않는다. 각 실행 State에 fingerprint와 프롬프트·입력·출력 해시를 함께 남긴다.

전체 출력이 달랐던 2개 사례(`high_liquidity_20`, `risk_override`)는 평가 필드가 모두
동일했으므로 차이는 Tax·Legal·Unique 자유서술 영역이다. 이 변동은 리스크 계산 입력에
사용하지 않으며 출력 해시로 감사한다.

## 재현성 보강 방식

1. gpt-4o Structured Output 스키마로 타입과 카테고리를 제한한다.
2. 고정 시나리오 필드 Age·Job·Goal·Asset·Risk를 Pydantic Literal로 재강제한다.
3. 명시적 원화 금액의 억 원 변환과 금액 기반 Liquidity 등급은 코드로 결정론화한다.
4. 프롬프트 버전, seed, 실제 배포명, 실제 응답 모델, system fingerprint, 입력·출력 해시를
   `ips_extraction_meta`에 기록한다.
5. 20사례 테스트셋은 CI에서 구조와 평가 로직을 검증하고, 실제 Azure 반복 평가는 키가 있는
   환경에서 릴리스 전 실행한다.
